The finance team needs a report page that can be filtered, plus a download feature that streams the currently-filtered rows as a spec-compliant CSV file. A working Qwik City project already exists with all dependencies installed, and the SQLite database is already seeded. Your job is to implement the report page and the CSV export endpoint. Everything runs locally; no external services are involved. ## Project & Data - Project path: `/home/user/qwik-app` - SQLite database file: `/home/user/qwik-app/data/reports.db` (already created and seeded; do NOT recreate or reseed it). - The database driver `better-sqlite3` is already installed and available. - Table `transactions` has exactly these columns: - `id` (INTEGER, primary key) - `date` (TEXT, formatted `YYYY-MM-DD`) - `category` (TEXT) - `description` (TEXT) - `amount` (REAL) ## Requirements Implement two routes in the Qwik City app: 1. A report page at `/reports` that: - Reads the current transactions from the SQLite database on the server (server-only) and renders them in the initial server-rendered HTML. - Honors three optional URL query parameters used as filters, combined with logical AND: - `from`: keep rows whose `date` is greater than or equal to this value (inclusive). - `to`: keep rows whose `date` is less than or equal to this value (inclusive). - `category`: keep rows whose `category` matches this value exactly (case-sensitive). - Renders each matching row's `description` text into the HTML. - Shows a link with the visible text `Download CSV` whose `href` points at the export endpoint below and carries the exact same `from`, `to`, and `category` filter values that are currently applied. 2. A CSV export endpoint at `/reports/export` (an HTTP GET handler) that: - Applies the same `from`, `to`, and `category` query-parameter filters, with identical semantics as the page. - Returns the filtered rows as a downloadable CSV file. ## Implementation Hints - Use a Qwik City `onGet` request handler for the export route so you fully control the HTTP response (headers and body). - Keep all database access inside server-only boundaries (route loaders and request handlers) so no database module leaks into the client bundle. - The response for the export endpoint must include these headers: - `Content-Type` whose value contains `text/csv`. - `Content-Disposition` set to attachment form and containing a filename that ends in `.csv` (use the filename `transactions.csv`). - The CSV body must follow RFC 4180: - The first line is a header row with exactly these column names in this order: `id,date,category,description,amount`. - One record per matching transaction, ordered by `id` ascending. - Records are separated by CRLF (`\r\n`). - Any field containing a comma, a double quote, or a newline must be enclosed in double quotes, and any double quote inside such a field must be escaped by doubling it (`"` becomes `""`). - The `amount` field is written as its plain numeric value (e.g. `12.5`, `40`, `100`). - Start command: `npm run dev` - Port: 5173 - Routes: - `GET /reports` (also accepts `from`, `to`, `category` query params): returns status 200 and an HTML page containing the filtered transaction descriptions and a `Download CSV` link. - `GET /reports/export` (also accepts `from`, `to`, `category` query params): returns status 200, a CSV attachment as described above.
