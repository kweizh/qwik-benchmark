# Qwik City Kanban Board with Drag-and-Drop Reordering and SQLite Persistence

## Background
Build a Kanban board application using Qwik City (the Qwik meta-framework). The board has three fixed columns — `todo`, `doing`, and `done` — each holding an ordered list of cards. Users move cards between columns and reorder them within a column using native HTML5 drag-and-drop. Every move is persisted to a local SQLite database so that the exact column and ordering survive a full page reload. The board is server-rendered from the database on every request.

## Requirements
- Render a board page at `/` with three columns (`todo`, `doing`, `done`), each showing its cards in persisted order.
- Cards must be movable between columns and reorderable within a column via native HTML5 drag-and-drop (wired with `useStore` and `onDragStart$` / `onDragOver$` / `onDrop$`).
- Persist every move to a local SQLite database, maintaining a stable, contiguous 0-based `position` index within each column.
- Server-render the board from the database using `routeLoader$` so ordering is correct after a reload (resumability / rehydration).
- Expose JSON HTTP endpoints for reading the board, applying a move, and resetting to the initial seed.
- Seed the database with the initial board state below when it is empty.

## Data model
Each card has: a stable integer `id`, a `title` (string), a `column` (`todo` | `doing` | `done`), and a `position` (0-based integer, contiguous within its column, ascending = top-to-bottom).

Initial seed (order top-to-bottom within each column):
- `todo`: id 1 "Design landing page", id 2 "Write unit tests", id 3 "Set up CI"
- `doing`: id 4 "Implement auth", id 5 "Refactor store"
- `done`: id 6 "Project scaffolding"

## Implementation Hints
- Use a Qwik City project. Keep all SQLite access strictly server-side (inside `routeLoader$` and server endpoints) so no database module leaks into the client bundle. `better-sqlite3` is a good local, container-friendly driver.
- Use `routeLoader$` to load the board from SQLite and server-render the columns; wire the drag-and-drop interactions on the client with `useStore` and `onDragStart$` / `onDragOver$` / `onDrop$`, calling the move endpoint to persist.
- When a card is moved, remove it from its source column and insert it into the target column at the target index, then re-number `position` values in every affected column so they stay contiguous (0..n-1) and ordering is preserved. Clamp an out-of-range target index to the end of the column.

- Project path: /home/user/qwik-app
- Start command: `npm run dev -- --port 3000 --host 0.0.0.0`
- Port: 3000

- Board page:
  - `GET /` returns server-rendered HTML. Each column container element must carry the attribute `data-column` with value `todo`, `doing`, or `done`. Each card element must carry `data-card-id="<id>"`, be draggable (have the `draggable` attribute), and display the card's title text. Cards must appear in persisted (position ascending) order within their column.

- HTTP JSON endpoints (requests and responses use `application/json`):
  - `GET /api/board`: returns status 200 and the current board:
    ```json
    {
      "todo":  [{ "id": number, "title": string, "position": number }],
      "doing": [{ "id": number, "title": string, "position": number }],
      "done":  [{ "id": number, "title": string, "position": number }]
    }
    ```
    Within each column, objects are ordered by `position` ascending, and `position` values are contiguous starting at 0.
  - `POST /api/move`: accepts a move, persists it, then returns status 200 with the full updated board (same shape as `GET /api/board`).
    ```json
    // Request
    { "cardId": number, "toColumn": "todo" | "doing" | "done", "toIndex": number }
    ```
    - Insert the card into `toColumn` at position `toIndex`; if `toIndex` is greater than the number of cards in the target column, append it at the end; a negative `toIndex` is treated as 0.
    - If `cardId` does not exist, return status 404. If `toColumn` is not one of the three valid columns, or if required fields are missing/invalid, return status 400.
  - `POST /api/reset`: restores the database to the exact initial seed above and returns status 200 with the seeded board (same shape as `GET /api/board`).
