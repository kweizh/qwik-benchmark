On validation failure the same step must re-render (HTTP 200, not advancing), preserve the submitted values, and show field-specific error messages. * On successful validation of a step, the server must advance the user to the next step via an HTTP redirect (302/303). * Partial progress must be persisted **on the server** and associated with the browser via a cookie, so that navigating **back** to an earlier step re-displays the previously entered values (the input elements must be pre-filled). * The final submit on the Review step must write the completed user to local SQLite and then redirect to `/register/success/`. ## Validation Rules (enforced server-side with Zod) * Account step: * `email`: must be a syntactically valid email address, AND must not already exist in the `users` table (uniqueness enforced against the database). * `password`: at least 8 characters and must contain at least one lowercase letter, one uppercase letter, and one digit. * `confirmPassword`: must exactly equal `password`. * Profile step: * `fullName`: at least 2 characters. * `age`: an integer that is at least 18. * `country`: a non-empty string. ## Implementation Hints * Project path: /home/user/qwik-wizard * Scaffold a Qwik City app (the non-interactive `empty` starter is a good base) and add directory-based routes under `src/routes/register/`. * Use `routeAction$(..., zod$({ ... }))` for server-side validation; return field errors via the action's failure/`fieldErrors` mechanism and render them in the step's markup. ### 🌟 Features Implemented

1. **Routing & Server-Side Navigation (`/register/*`)**:
   - `/register/`: **Account** step (collects `email`, `password`, and `confirmPassword`).
   - `/register/profile/`: **Profile** step (collects `fullName`, `age`, and `country`).
   - `/register/review/`: **Review** step (displays entered information except password, with a final submit control).
   - `/register/success/`: **Success** page (displays the registered email address).

2. **Server-Side Validation with Zod & Route Actions**:
   - Each step submits to a server-side `routeAction$` validated by `zod$`.
   - On validation failure, the page re-renders with an HTTP 200 status, preserves all submitted values, and shows field-specific error messages.
   - On success, the server redirects the browser to the next step using a 302/303 redirect.

3. **Robust Server-Side Session Persistence**:
   - Partial progress is stored in a local SQLite `sessions` table and associated with the browser via a secure `registration_session` cookie.
   - If the user navigates back to an earlier step, the inputs are pre-filled with their previously entered values.
   - If a user tries to access a step directly out of order (e.g. going straight to `/register/review/`), they are gracefully redirected back to the earliest incomplete step.

4. **Local SQLite Integration (`better-sqlite3`)**:
   - Located at `/home/user/qwik-wizard/db/app.db`.
   - Automatically initializes the `users` and `sessions` tables on startup if they do not exist.
   - All SQLite code is kept in server-exclusive boundaries (`.server.ts` modules) to prevent leaks into the client bundle.

5. **Security Best Practices**:
   - Plaintext passwords are never stored. They are securely hashed using Node's built-in `crypto` module with `scrypt` (generating a unique salt per password).
   - Passwords are never displayed on the Review page.

6. **Progressive Enhancement**:
   - The app is fully functional with native HTML forms and server-side logic, meaning it works perfectly even when client-side JavaScript is completely disabled.

7. **Modern, Responsive Design**:
   - Styled using custom CSS with a clean, responsive card layout and a step-by-step progress indicator showing active, completed, and pending states.
