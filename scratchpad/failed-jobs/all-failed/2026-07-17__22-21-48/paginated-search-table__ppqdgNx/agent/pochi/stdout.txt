## What was built

**1. Server-only SQLite module — `src/server/db.ts`**
- Creates a singleton in-memory SQLite database and loads the canonical dataset from `data/products.json` (read relative to `process.cwd()`).
- Exposes `getDb()`. It's only ever imported dynamically from inside the `routeLoader$` body, so `better-sqlite3` (a native addon) never leaks into the client bundle.

**2. Route + loader — `src/routes/products/index.tsx`**
- `routeLoader$` (server-only) reads `q`, `sort`, `dir`, `page`, `pageSize` from the request URL with the required defaults and validation (sort whitelist, dir ∈ asc/desc, page/pageSize clamped to positive integers).
- Queries SQLite with `WHERE name LIKE ?` (case-insensitive substring), `ORDER BY <col> <dir>` (column from a fixed whitelist → safe interpolation), `LIMIT/OFFSET`, plus a separate `COUNT(*)` for the total.
- Returns `{ rows, total, page, pageSize, totalPages, q, sort, dir }`.

**3. UI**
- HTML `<table>` rendering each row's `name`, `category`, `price`, `stock` (names appear verbatim).
- `Total: <n>` and `Page <page> of <totalPages>` rendered as single contiguous template expressions so the exact strings survive in the raw HTML (Qwik's hydration comment markers don't split them).
- Sortable column headers (Name/Category/Price/Stock) as `<Link>`s; clicking the active column toggles `dir`, others set `sort` with `dir=asc` (and reset to page 1).
- Previous/Next pagination controls that update the `page` query param (disabled at the bounds).
- A debounced (~300ms) search `<input>` that updates the `q` query param (and resets to page 1) via `useNavigate`, making the URL the single source of truth.
